Implementation plan

Pi TUI for the RLM harness

A lightweight Pi-style terminal UI with multiline input, inline transcript, concise REPL traces, and a clean harness/UI boundary.

Source: docs/work/002-pi-tui-rlm-interface/research.md Plan: docs/work/002-pi-tui-rlm-interface/plan.md Status: draft for review

01 · Problem

Current CLI is too thin for RLM UX

The readline loop mixes input, history, traces, and agent orchestration, and cannot support a comfortable trace-rich terminal experience.

Need

Multiline composer, inline chronological transcript, clear final answers, and trace/tool visibility.

Constraint

Agent and REPL layers must remain independent from TUI packages and terminal concepts.

Mode

Auto TUI for interactive terminals, minimal no-TUI for --no-tui and non-TTY.

Certainty

Behavior must be verifiable through unit, render-model, fake-terminal, and PTY tests.

02 · Architecture

Three layers, one session contract

flowchart LR
  Main[main.ts] --> Args[cli args]
  Main --> Mode{TUI?}
  Mode -->|yes| PiApp[Pi TUI app]
  Mode -->|no| Plain[no-TUI runner]
  PiApp --> Session[RlmSession]
  Plain --> Session
  Session --> Runner[AgentRunner]
  Runner --> Agent[runAgentTurn]
  Agent --> Repl[REPL/tools]
  PiApp --> PiTui[pi-tui primitives]
Harness: no TUI imports Session: history + events UI: render + input only

03 · Overall acceptance

Programmatically verifiable done criteria

npm install npm run typecheck npm test npm run build npm run test:pty
  • @earendil-works/pi-tui, node-pty, and Node >=22.19.0 are declared in package metadata and lockfile.
  • Clean build removes stale dist/tui/* artifacts from the previous implementation attempt.
  • Dependency-boundary tests prove harness/agent/REPL modules do not import Pi TUI.
  • PTY tests validate critical UX without Anthropic, Pyodide, network, or ANSI snapshots.

04 · Key interfaces

AgentRunner + RlmSession

AgentRunner

Injected boundary wrapping runAgentTurn(history, input, { onTrace }). UI code never calls AI SDK or Pyodide directly.

RlmSession

Owns history, busy state, command recognition, trace forwarding, final/error events, and deterministic IDs/clocks for tests.

SubmitResult

accepted, ignored: empty, rejected: busy, or exit.

Session events

turn-started, trace, turn-final, turn-error, turn-finished.

05 · UI model

Inline transcript with global filters

The first pass keeps state simple: user item, trace group, final answer/error, with toggles that reveal preserved trace details.

  • Default filters hide assistant trace, reasoning trace, and full tool details.
  • Concise REPL entries show tool name, code summary, status, and compact output markers.
  • Ctrl+T, Ctrl+A, Ctrl+R toggle tool details, assistant trace, and reasoning.
  • Every custom render line must satisfy visibleWidth(line) <= width.

07 · Milestones 1–2

Foundation and session bubble

M1: Dependency/build/CLI foundation

Add Pi TUI dependencies, Node engine update, clean build, and mode parser.

M2: Session/controller

Add AgentRunner and RlmSession for history, events, commands, errors, and busy state.

Acceptance focus

  • Arg tests cover auto, forced modes, conflicts, unknown flags, and TTY checks.
  • Session tests cover success, trace order, errors, busy rejection, empty input, CRLF, and exit commands.
  • Boundary test protects agent/REPL modules from TUI imports.

08 · Milestones 3–4

Plain mode and pure UI state

M3: No-TUI runner

Move plain behavior into src/cli/no-tui.ts using the same session contract.

M4: TUI state + trace log

Map trace events into deterministic display entries and reducer-managed transcript state.

Acceptance focus

  • No-TUI tests cover interactive success, errors, /exit, and non-TTY one-shot stdin.
  • Trace-log tests cover all AgentTraceEvent variants and REPL status classification.
  • State tests prove filters hide by default without losing detail data.

09 · Milestones 5–6

Components and app integration

M5: TUI components

Transcript, trace blocks, status line, Markdown final answers, and Editor-based composer.

M6: Pi TUI app wiring

Create runPiTuiApp, connect session events to state, and integrate with main.ts.

Acceptance focus

  • Render tests assert line width at 18, 40, and 100 columns.
  • Collapsed/expanded REPL, assistant trace, reasoning, and Markdown behavior are tested.
  • Forced TUI on non-TTY fails clearly; auto mode falls back to no-TUI.

10 · Milestone 7

11 · User verification

How reviewers can try it

npm run build node dist/main.js --no-tui node dist/main.js --tui
  • Default interactive launch opens the TUI.
  • --no-tui keeps a simple prompt and plain trace rendering.
  • Transcript distinguishes user, final answer, trace, REPL, and errors.
  • /exit, /quit, and idle Ctrl+C exit cleanly.

12 · Invariants

What must not change?

These constraints are regression guards throughout implementation.

Harness isolation

No Pi TUI imports in agent, trace, REPL, or tool modules.

RLM semantics

Final answers still come only from FINAL/FINAL_VAR.

No-TUI simplicity

Plain mode does not duplicate TUI filters or state.

No fake production path

Fake agent behavior lives only in test fixtures.

13 · Stop-and-ask

Pause if these assumptions break

  • Pi TUI cannot install/typecheck under Node >=22.19.0.
  • Installed Editor behavior differs from researched submit/newline/backslash-enter behavior.
  • Ctrl+Enter requires invasive keybinding changes beyond a local wrapper.
  • Clean shutdown while a turn runs requires adding cancellation to runAgentTurn.
  • PTY tests require more new dependencies than approved node-pty.
If none of these boundaries are crossed, the plan is ready for autonomous implementation with TDD.
